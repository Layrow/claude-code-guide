# 检查点与回退

检查点让你可以保存会话状态，并在 Claude Code 会话中回退到之前的某个时刻。这对于探索不同方案、从错误中恢复，或对比多个备选解法都非常有价值。

## 概览

检查点允许你保存会话状态，并回退到之前的时间点，从而安全地进行试验和多路径探索。它本质上是你当前会话状态的快照，其中包括：
- 所有往返消息
- 已进行的文件修改
- 工具使用历史
- 会话上下文

当你需要尝试不同方案、从失误中恢复，或比较不同实现时，检查点会非常有用。

## 核心概念

| 概念 | 说明 |
|---------|-------------|
| **Checkpoint（检查点）** | 会话状态的快照，包含消息、文件和上下文 |
| **Rewind（回退）** | 返回到某个先前检查点，并丢弃之后的变更 |
| **Branch Point（分支点）** | 用来从同一起点探索多种方案的检查点 |

## 如何访问检查点

你主要可以通过两种方式访问和管理检查点：

### 使用快捷键
连续按两次 `Esc`（`Esc` + `Esc`）即可打开检查点界面，并浏览已保存的检查点。

### 使用 Slash Command
使用 `/rewind` 命令（别名：`/checkpoint`）可以快速访问：

```bash
# 打开回退界面
/rewind

# 或使用别名
/checkpoint
```

## 回退选项

当你执行回退时，会看到一个包含五个选项的菜单：

1. **Restore code and conversation** -- 同时恢复文件和对话到该检查点
2. **Restore conversation** -- 只回退消息，保留当前代码不变
3. **Restore code** -- 只恢复文件变更，保留完整对话历史
4. **Summarize from here** -- 不直接丢弃此后的对话，而是从这个点开始压缩为 AI 生成的摘要。原始消息仍会保留在转录中。你还可以额外提供说明，让摘要聚焦特定主题。
5. **Never mind** -- 取消并返回当前状态

## 自动检查点

Claude Code 会自动为你创建检查点：

- **每次用户输入** - 每次用户发出新提示时，都会自动创建一个新的检查点
- **持久保存** - 检查点会跨会话保留
- **自动清理** - 检查点会在 30 天后自动清理

这意味着你始终可以回退到对话中的任意一个先前时刻，无论是几分钟前还是几天前。

## 使用场景

| 场景 | 工作流 |
|----------|----------|
| **探索不同方案** | 保存 → 试方案 A → 保存 → 回退 → 试方案 B → 对比 |
| **安全重构** | 保存 → 重构 → 测试 → 如果失败：回退 |
| **A/B 测试** | 保存 → 设计 A → 保存 → 回退 → 设计 B → 对比 |
| **从错误中恢复** | 发现问题 → 回退到上一个可用状态 |

## 使用检查点

### 查看与回退

连续按两次 `Esc` 或使用 `/rewind` 打开检查点浏览器。你会看到所有可用检查点及其时间戳列表。选择任意检查点即可回退到对应状态。

### 检查点详情

每个检查点都会展示：
- 创建时间戳
- 被修改过的文件
- 对话中的消息数量
- 使用过的工具

## 实战示例

### 示例 1：探索不同方案

```
User: Let's add a caching layer to the API

Claude: 我来给你的 API 端点加入 Redis 缓存……
[在检查点 A 进行了修改]

User: Actually, let's try in-memory caching instead

Claude: 我来回退一下，换一种方案探索……
[用户按下 Esc+Esc 并回退到检查点 A]
[在检查点 B 实现内存缓存]

User: Now I can compare both approaches
```

### 示例 2：从错误中恢复

```
User: Refactor the authentication module to use JWT

Claude: 我来把认证模块重构为使用 JWT……
[进行了大范围修改]

User: Wait, that broke the OAuth integration. Let's go back.

Claude: 我来帮你回退到重构之前……
[用户按下 Esc+Esc，并选择重构前的检查点]

User: Let's try a more conservative approach this time
```

### 示例 3：安全试验

```
User: Let's try rewriting this in a functional style
[在试验前创建检查点]

Claude: [进行试验性修改]

User: The tests are failing. Let's rewind.
[用户按下 Esc+Esc 并回退到该检查点]

Claude: 我已经把这些修改回退了。我们换一种方式试试。
```

### 示例 4：分支式探索

```
User: I want to compare two database designs
[记下当前检查点，可以叫它“Start”]

Claude: 我先实现第一个设计……
[实现 Schema A]

User: Now let me go back and try the second approach
[用户按下 Esc+Esc 并回退到 “Start”]

Claude: 现在我来实现 Schema B……
[实现 Schema B]

User: Great! Now I have both schemas to choose from
```

## 检查点保留策略

Claude Code 会自动管理你的检查点：

- 每次用户输入都会自动创建检查点
- 旧检查点最长保留 30 天
- 系统会自动清理检查点，避免存储无限增长

## 工作流模式

### 用于探索的分支策略

当你要探索多种方案时：

```
1. 从初始实现开始 → 检查点 A
2. 尝试方案 1 → 检查点 B
3. 回退到检查点 A
4. 尝试方案 2 → 检查点 C
5. 比较 B 和 C 的结果
6. 选择最佳方案并继续
```

### 安全重构模式

当你要做较大的修改时：

```
1. 当前状态 → 检查点（自动）
2. 开始重构
3. 运行测试
4. 如果测试通过 → 继续工作
5. 如果测试失败 → 回退并改用其他方案
```

## 最佳实践

由于检查点会自动创建，你可以把注意力放在工作本身，而不用担心手动保存状态。不过，下面这些实践仍然值得记住：

### 高效使用检查点

✅ **建议这样做：**
- 回退前先查看有哪些可用检查点
- 当你想尝试不同方向时，使用回退
- 保留检查点，方便比较不同方案
- 先理解每种回退选项的作用（恢复代码和对话、恢复对话、恢复代码、从这里开始摘要）

❌ **不建议这样做：**
- 只依赖检查点来保存代码
- 认为检查点会跟踪外部文件系统变化
- 把检查点当成 git commit 的替代品

## 配置

你可以在设置中切换自动检查点：

```json
{
  "autoCheckpoint": true
}
```

- `autoCheckpoint`: 是否在每次用户输入时自动创建检查点（默认：`true`）

## 限制

检查点存在以下限制：

- **不会跟踪 Bash 命令造成的更改** - 像 `rm`、`mv`、`cp` 这类文件系统操作不会被检查点捕获
- **不会跟踪外部改动** - 在 Claude Code 之外做的修改（例如在编辑器、终端里）不会被捕获
- **不能替代版本控制** - 想要永久、可审计地保存代码变更，还是要用 git

## 故障排查

### 找不到检查点

**问题**：预期的检查点没有出现

**解决方案**：
- 检查是否清理过检查点
- 确认设置中 `autoCheckpoint` 已启用
- 检查磁盘空间

### 回退失败

**问题**：无法回退到检查点

**解决方案**：
- 确认没有冲突的未提交变更
- 检查检查点是否损坏
- 尝试回退到其他检查点

## 与 Git 的配合

检查点可以补充 git，但不能替代 git：

| 功能 | Git | Checkpoints |
|---------|-----|-------------|
| 范围 | 文件系统 | 对话 + 文件 |
| 持久性 | 永久 | 基于会话 |
| 粒度 | Commit | 任意时刻 |
| 速度 | 较慢 | 即时 |
| 可共享性 | 是 | 有限 |

建议二者搭配使用：

1. 使用检查点进行快速试验
2. 使用 git commit 保存最终确认的修改
3. 在执行 git 操作前确认当前检查点状态
4. 将成功的检查点状态提交到 git

## 快速上手指南

### 基本流程

1. **正常工作** - Claude Code 会自动创建检查点
2. **想回到之前？** - 连按两次 `Esc` 或使用 `/rewind`
3. **选择检查点** - 从列表中选择要回退到的节点
4. **选择恢复内容** - 可选恢复代码和对话、恢复对话、恢复代码、从这里开始摘要或取消
5. **继续工作** - 你就回到了那个时间点

### 快捷键

- **`Esc` + `Esc`** - 打开检查点浏览器
- **`/rewind`** - 访问检查点的另一种方式
- **`/checkpoint`** - `/rewind` 的别名

## 何时该回退：上下文监控

检查点能让你回到过去，但你怎么知道 *什么时候* 该回退？随着对话不断增长，Claude 的上下文窗口会逐渐被占满，模型质量也会悄悄下降。你可能在不知不觉中，让一个“视野已经变窄”的模型继续产出代码。

**[cc-context-stats](https://github.com/luongnv89/cc-context-stats)** 通过在 Claude Code 状态栏中加入实时的 **上下文分区** 来解决这个问题。它会告诉你当前处在上下文窗口的哪个阶段：从 **Plan**（绿色，适合规划和编码），到 **Code**（黄色，不建议再开启新的计划），再到 **Dump**（橙色，该收尾并回退了）。当你看到分区发生变化，就知道该打一个检查点并重新开始，而不是继续在输出质量下降的状态下硬撑。

## 总结

检查点是 Claude Code 的一个自动功能，它让你能够放心探索不同方案，而不用担心丢失已有工作。每次用户输入都会自动创建一个新的检查点，因此你可以回退到会话中的任意一个先前时刻。

核心收益：
- 无压力地试验多种方案
- 快速从错误中恢复
- 并排比较不同解法
- 与版本控制系统安全配合

记住：检查点不是 git 的替代品。用检查点做快速试验，用 git 保存长期有效的代码变更。
