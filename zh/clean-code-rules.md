# 面向 AI 代码生成的整洁代码规则

这些规则用于指导代码生成，使产出的代码更易维护，也更符合专业级质量要求。

## 有意义的命名
- 使用能够体现意图的命名，让人一眼看出它为什么存在
- 避免误导性命名和没有意义的区分（例如 `data`、`info`、`manager`）
- 使用可读、可搜索、可发音的命名
- 类名：使用名词（例如 `UserAccount`、`PaymentProcessor`）
- 方法名：使用动词（例如 `calculateTotal`、`sendEmail`）
- 避免让人做额外脑内映射，也不要使用编码式命名（如 Hungarian notation、前缀）

## 函数
- 保持函数短小（理想上少于 20 行）
- 只做一件事，遵循单一职责原则
- 每个函数只保持一个抽象层级
- 控制参数数量：理想是 0-2 个，最多 3 个，避免 flag 参数
- 不要产生副作用，函数应当名副其实
- 将命令（修改状态）与查询（返回信息）分开
- 优先使用异常，而不是错误码

## 注释
- 代码应尽量自解释，能不写注释就不写
- 好的注释：法律信息、警告、TODO、公有 API 文档
- 坏的注释：重复、误导，或只是解释糟糕代码
- 永远不要注释掉代码，直接删除即可（版本控制会保留历史）
- 如果你不得不写注释，先考虑是不是应该重构代码

## 格式
- 保持文件短小、聚焦
- 纵向排版：相关概念放在一起，用空行分隔不同概念
- 横向排版：控制行长度（80-120 字符）
- 使用一致的缩进和团队风格
- 将相关函数组织在一起

## 对象与数据结构
- 对象：通过抽象隐藏数据，通过方法暴露行为
- 数据结构：暴露数据，行为尽量少
- 迪米特法则：只和直接朋友说话，避免 `a.getB().getC().doSomething()`
- 不要不加思考地通过 getter/setter 暴露内部结构

## 错误处理
- 使用异常，而不是返回码或错误标志
- 可能失败的代码，优先先写 `try-catch-finally`
- 在异常消息中提供上下文
- 不要返回 `null`，返回空集合或使用 Optional/Maybe
- 不要把 `null` 作为参数传递

## 类
- 小类的衡量标准是职责，而不是代码行数
- 单一职责原则：一个类只应该有一个变化原因
- 高内聚：类中的变量应被多个方法共同使用
- 低耦合：类之间的依赖尽量少
- 开闭原则：对扩展开放，对修改封闭

## 单元测试
- 快速、独立、可重复、自校验、及时（F.I.R.S.T.）
- 每个测试一个断言（或一个核心概念）
- 测试代码质量应与生产代码质量一致
- 测试名称要可读，并说明测试内容
- 使用 Arrange-Act-Assert 模式

## 代码质量原则
- **DRY（Don't Repeat Yourself）**：避免重复
- **YAGNI（You Aren't Gonna Need It）**：不要为假设中的未来提前设计
- **KISS（Keep It Simple）**：避免不必要的复杂度
- **Boy Scout Rule**：离开时让代码比你来时更整洁

## 需要避免的代码坏味道
- 过长的函数或类
- 重复代码
- 死代码（未使用的变量、函数、参数）
- Feature envy（方法更关心别的类，而不是自己所在的类）
- Inappropriate intimacy（类之间知道彼此太多内部细节）
- 过长的参数列表
- Primitive obsession（过度使用基础类型，而不是小对象）
- Switch/case 语句过多（考虑使用多态）
- Temporary fields（类变量只在少数情况下使用）

## 并发
- 将并发代码与其他代码分离
- 限制被 synchronized/locked 数据的作用范围
- 使用线程安全集合
- 保持同步区尽量小
- 理解你的执行模型和并发原语

## 系统设计
- 将构造与使用分离（依赖注入）
- 对复杂对象创建使用 factory、builder
- 面向接口编程，而不是面向实现
- 组合优于继承
- 只在真正能简化设计时使用设计模式，不要为了炫技

## 重构
- 持续重构，而不是一次性大重构
- 重构前后都要确保测试通过
- 小步前进：一次只做一个改动
- 常见重构：Extract Method、Rename、Move、Inline

## 文档
- 自解释代码 > 注释 > 外部文档
- 公有 API 需要清晰文档
- 在文档中加入示例
- 文档尽量靠近代码（理想情况是就在代码里）

---

**核心理念**：代码被阅读的次数远高于被编写的次数。优化目标应该是可读性和可维护性，而不是“聪明感”。
